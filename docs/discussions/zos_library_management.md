# z/OS 運用資材の軽量ライブラリ管理ツール

> Excel/VBA + 3270 エミュレータによる Change Tracker 軽量代替の検討
> 2026 年 4 月 / IBM 公式マニュアル 21 出典に基づく検討資料

[スライド版 PDF をダウンロード](../assets/discussions/zos_library_management_full_v1.pdf){ .md-button .md-button--primary }

!!! abstract "本資料の趣旨"
    z/OS の運用資材（運用部門が日常編集する JCL / REXX / PROC 等）について、公式ツール (z/OS Change Tracker) を導入できない小規模現場での軽量代替案を提案する。背景・対象範囲・選択肢比較・本ツールの守備範囲・残された判断を順に説明する。

---

## エグゼクティブサマリ

### 目的

z/OS の運用資材（運用部門が日常編集する JCL / REXX / PROC 等）について、変更履歴を追跡し過去版を再現できる軽量管理ツールを構築できるかを判定する。

### 結論

Excel/VBA + 3270 エミュレータ (PCOMM) のみで運用資材の変更管理が成立する。Change Tracker の全 48 機能のうち **32 機能を等価実装**し、**16 機能を割り切る**。z/OS 側に常駐プログラム・追加ライセンスを置かない設計で、Change Tracker を導入できない小規模現場の管理空白を埋める。

### 主要判断ポイント（4 件）

1. 対象は z/OS の 6 領域のうち**運用資材領域 1 つに絞る**（OS 担当者管轄の 5 領域は対象外）
2. Change Tracker は機能比較の参照対象であり、本ツールの採否対象ではない（採用しないことは前提）
3. データ取得は PCOMM の IND$FILE による Shift-JIS テキスト転送に限定（REST/FTP/SSH/Zowe は使わない）
4. 差分検出はファイル 1 つ単位のテキスト比較に限定（バイナリ・ボリューム単位は対象外）

## 本資料の前提（SSOT）— 検討開始前の既決事項

本資料では議論しない、6 つの前提条件。

| # | 項目 | 内容 |
|---|---|---|
| ① | 実装方式 | Excel/VBA + 3270 エミュレータのみで実装。z/OS 側に常駐プログラム・データセット・追加ライセンスを置かない。 |
| ② | Change Tracker | IBM 公式の z/OS Change Tracker は採用しない（priced feature + STC 常駐 + RACF プロファイル設計が必要で、想定組織には合わない）。本資料では機能比較のベンチマークとして参照する。 |
| ③ | 対象範囲の判定基準 | 業務担当者が日常的に作成・修正するファイルか / OS 担当者が管理するシステムファイルかで切り分ける（役割境界による分類）。 |
| ④ | 対象環境の前提 | 対象 z/OS 環境には USS (UNIX System Services) を導入していない。USS 関連ファイルは検討対象としない。 |
| ⑤ | データ取得経路 | IBM Personal Communications (PCOMM) の IND$FILE による Shift-JIS テキスト転送に限定。REST/FTP/SSH/Zowe は対象外。 |
| ⑥ | 差分検出の粒度 | ファイル 1 つ単位でテキスト比較（JCL 1 本、パラメータ 1 ファイル等）。コンパイル後バイナリ（ロードモジュール）とディスクボリューム全体は対象外。 |

## 全体観: z/OS のライブラリは 6 領域に分かれる

本ツールの対象は 1 領域（運用資材）のみ、他 5 領域は対象外。

z/OS のライブラリは、所有者と公式管理体系で 6 領域に分けて整理できる。本ツールが対象とするのは「運用資材領域」1 つのみ。他 5 領域は OS 担当者の管轄であり、それぞれに既存の公式管理体系（SMP/E、SCM 等）があるため対象外とする。

| 領域名（本資料分類） | 所有者 | 公式管理体系 | 本ツール扱い | 代表的なファイル |
|---|---|---|---|---|
| IBM 保守領域 | OS 担当 | SMP/E | 対象外 | IPL コア (SYS1.NUCLEUS 等) |
| システム制御領域 | OS 担当 | 運用ルール | 対象外 | PARMLIB (IEASYSxx, SMFPRMxx 等) |
| システムプロシージャ領域 | OS 担当 | 運用ルール | 対象外 | SYS1.PROCLIB の標準 PROC |
| アプリ実行領域 | アプリ部門 | SCM + リリース手順 | 対象外 | USER.LOADLIB (RECFM=U バイナリ) |
| アプリソース領域 | アプリ部門 | SCLM/Endevor/Git | 対象外 | アプリソース、JCL |
| **運用資材領域** | **運用部門** | **（公式の体系なし）** | **★ 本ツールの対象 ★** | **運用 JCL/REXX/PROC、運用所管 PARMLIB** |

## 運用資材領域には公式の管理体系が無い

既存ツールはいずれも運用資材のメンバ単位差分を体系的に追跡しない。

| 既存ツール | 本来の管轄 | 運用資材領域への適用上の限界 |
|---|---|---|
| SMP/E | IBM 保守領域 | 運用資材は SMP/E の対象範囲外 |
| SCM (SCLM/Endevor/Git) | アプリソース領域 | アプリ部門の管轄。運用資材まで全面拡張する案は別検討 |
| RACF / SMF | アクセス監査 | 「誰が・いつ」までは追えるが「中身が何に変わったか」は追えない |
| ISPF スタッツ | メンバ単位の更新追跡 | RECFM=U には付かない / Reset Statistics で改竄可能 |
| z/OS Change Tracker (V2R5+) | 公式の唯一の解 | STC 常駐 + VSAM repository + RACF プロファイル設計 + priced feature。小規模現場では導入要件を満たせない |

!!! note "帰結"
    運用資材領域は公式管理空白。Change Tracker のみが公式の解だが、小規模現場では導入要件（priced feature・STC 常駐・RACF プロファイル）を満たせない。
    → Change Tracker を導入できない現場のための、**軽量な代替手段が必要**。

## 本ツールの位置づけ

Change Tracker の劣化版でも上位互換でもない。**対象顧客が違う別物**。

| 軸 | z/OS Change Tracker | 本ツール (Excel/VBA + 3270) |
|---|---|---|
| 対象組織規模 | 中〜大規模（専任 z/OS 担当あり） | 小規模（利用者単独〜5 人未満、専任なし） |
| 導入要件 | STC + VSAM repository + RACF プロファイル + priced feature | Excel/VBA + PCOMM のみ |
| z/OS 側設置物 | 常駐プログラム・データセット・ライセンス | なし |
| 対応機能数 | 48 機能（フルセット） | 32 機能を等価実装、16 機能を割り切る |
| 対象データ形式 | PDS/PDSE バイナリ・テキスト両方、VSAM、ボリューム単位 | PDS/PDSE のメンバ単位テキストのみ |
| 差分検出のリアルタイム性 | STC 常駐によるリアルタイム監視可能 | PCOMM 経由のスナップショット差分のみ |
| 想定運用 | 監査要件のある中〜大規模システム | 公式管理空白を埋める軽量運用 |

!!! success "結論"
    本ツールは Change Tracker の代替品ではなく、Change Tracker を導入できない現場のための**別製品**。

## §A. 6 領域の判定詳細

### A-1. OS 担当者管轄の 5 領域はすべて対象外

| 領域名 | 重要度 | 対象範囲 | 判定 | 判定根拠 |
|---|---|---|---|---|
| IBM 保守領域 | 高 | z/OS の起動時に必須のシステムライブラリ群 (SYS1.NUCLEUS 等) | 本ツール対象外。SMP/E に管理を委ねる | 公式マニュアルで「IPL 時に決定され never be changed or removed」と明記 (S2) |
| システム制御領域 | 中 | z/OS の動作を制御する設定パラメータ群 (PARMLIB の IEASYSxx, SMFPRMxx 等) | 本ツール対象外。ただし運用所管メンバ (COMMNDxx 等) は運用資材領域に含める | Common Criteria (S13) で SMFPRMxx が監査対象と明記 |
| システムプロシージャ領域 | 中 | z/OS のシステム標準プロシージャ群 (SYS1.PROCLIB) | 本ツール対象外。運用カスタム PROC は運用資材領域に含める | JES2 起動時に決定される構造 (S3, S4) |

### A-2. アプリ部門領域 + 運用資材領域

| 領域名 | 重要度 | 対象範囲 | 判定 | 判定根拠 |
|---|---|---|---|---|
| アプリ実行領域 | 中 | アプリのロードモジュール (USER.LOADLIB) | 本ツール対象外。バイナリ差分は L5 (アプリソース) で追跡することで補える | RECFM=U で人間可読でなく、Excel で差分を見ても理解不能 (S6, S11) |
| アプリソース領域 | 中 | アプリ部門のソースコードと JCL | 本ツール対象外。既存 SCM に管理を委ねる | SCLM Project Manager Guide (S8) で SCM 中核領域として定義 |
| **運用資材領域 ★** | 高 | 運用部門が直接編集する JCL / REXX / PROC 等。運用所管 PARMLIB メンバ (COMMNDxx 等) を含む | **本ツールの唯一の対象**。Change Tracker の代替として軽量版を構築 | S5 / S13 / S17 で公式管理空白を確認。RACF/SMF (S13) は中身差分を追えず |

!!! info "§A の総括"
    OS 担当者管轄（5 領域）はすべて公式管理体系がある or 触らない。運用部門管轄の運用資材領域だけが公式管理空白で、本ツールが埋める対象。

## §B. 既存技術の限界

### B-1. ISPF スタッツでは追えない

| 限界 | 内容 | 本ツールへの含意 | 出典 / 根拠 |
|---|---|---|---|
| RECFM=U には付かない | ロードモジュール (RECFM=U) には ISPF スタッツが付与されない | アプリ実行領域の変更追跡には使えない | 公式メッセージ zOS31_f54mc00 (S11) |
| Reset Statistics で改竄可能 | ISPF Edit 上で手動で消去・改竄が可能 | 「最終更新者・最終更新時刻」が信頼できない | 公式マニュアル zOS31_f54u200 (S10) |
| 「何を変えたか」は追えない | 更新者・時刻のメタデータのみ。中身の差分は記録されない | 「3 日前の中身は何だったか」が再現できない | ISPF メタデータ仕様 (S10, S11) |

!!! info "§B-1 の含意"
    ISPF スタッツは「誰が・いつ」の補助的指標としては有用だが、変更管理の主要メカニズムにはならない。本ツールは ISPF スタッツに頼らず、PCOMM 経由でメンバ本体を取得し、Excel 側で完全な差分履歴を保持する設計とする。

### B-2. 本ツールの取扱単位は PDS/PDSE のメンバ

VSAM・USS など他形式は対象外として明示する。

| 形式 | 概要 | 本ツール扱い |
|---|---|---|
| PDS | Partitioned Data Set。固定形式の区分データセット。メンバ単位で読み書き可 | 対象（テキストのみ） |
| PDSE | PDS の内部構造改良版。メンバ単位読み書きは PDS と同じ | 対象（テキストのみ） |
| VSAM | キー付きレコード形式。PDS とは構造が異なり、メンバ概念がない | 対象外 (T22) |
| USS | UNIX System Services 上のファイル (zFS、HFS) | 対象環境に存在しないため対象外 (SSOT ④) |
| ボリューム単位 | VOLSER 単位で取得（数 GB 規模） | 3270 経由では数時間〜数日要する (T24) ため対象外 |
| RECFM=U | ロードモジュール等の不定長レコード。3270 で取得しても可読テキストにならない | 対象外 (T23) |

## §C. 機能群 8 群の詳細判定

### A 群: 対象指定 / 保護管理 (T9)

**対象機能:** Change Tracker (S15) の PROTECT / EXPLICIT / PATTERN / EXCLUDE 4 機能
**論点:** 対象 DSN・パターン・除外指定を Excel ワークシートで等価実装可能か

| # | 機能 | 判定 | 実装方法 / 割切理由 |
|---|---|---|---|
| A1 | PROTECT (保護対象指定) | 採用 | Excel の対象 DSN 列に明示。VBA で対象判定 |
| A2 | EXPLICIT (明示的対象指定) | 採用 | PROTECT と同列で実装 |
| A3 | PATTERN (パターン指定) | 採用 | Excel ワイルドカード列を追加 |
| A4 | EXCLUDE (除外指定) | 採用 | Excel 除外列を追加 |

→ 4 機能すべて採用。Excel ワークシートに対象 DSN・パターン・除外列を設計。

### B 群: バックアップ / 世代管理 (T10)

**対象機能:** Change Tracker (S15) のバックアップ / 世代管理 9 機能
**論点:** リアルタイム監視（STC 常駐）を諦めても本ツールの中核機能（履歴を残す）が成立するか

| # | 機能 | 判定 | 実装方法 / 割切理由 |
|---|---|---|---|
| B1 | RT (リアルタイム監視) | 割切 | STC 常駐を要するため本ツールでは不採用 |
| B2 | MEMBERBACKUP (メンババックアップ) | 採用 | Excel 別シートに過去版を保管 |
| B3 | AUTOARC (自動アーカイブ) | 採用 | VBA で世代切替 |
| B4 | COPIES (世代数指定) | 採用 | Excel 設定列で世代数指定 |
| B5 | DAYS (日数指定) | 採用 | Excel 設定列で日数指定 |
| B6 | FORCE (強制バックアップ) | 採用 | VBA ボタンで明示実行 |
| B7 | VERIFY (検証) | 採用 | Excel チェックサム列 |
| B8 | ARCHIVE (アーカイブ) | 採用 | 別ブック分離による長期保管 |
| B9 | SCHEDULED (定期実行) | 採用 | Windows タスクスケジューラ連携 |

→ 9 機能のうち 8 機能採用。リアルタイム監視のみ割切（PCOMM スナップショット方式に変更）。

### C 群: 比較 / 差分検出 (T11)

**対象機能:** Change Tracker (S15) の比較 / 差分検出 7 機能
**論点:** 「3 日前の中身は何か」を保ちつつ、技術制約（3270 経由 / バイナリ取得困難）と整合する選択をする

| # | 機能 | 判定 | 実装方法 / 割切理由 |
|---|---|---|---|
| C1 | テキスト diff | 採用 | VBA + Diff ライブラリ |
| C2 | 複数環境比較（本番/開発） | 採用 | Excel 比較シートで対応 |
| C3 | 任意 2 版比較 | 採用 | Excel 履歴シート任意行同士で比較 |
| C4 | 時系列詳細表示 | 採用 | Excel ピボット |
| C5 | バイナリ diff（ロードモジュール） | 割切 | RECFM=U は人間可読でないため (T23) |
| C6 | ボリューム単位比較 | 割切 | 3270 経由では時間的に困難 (T24) |
| C7 | USS 配下比較 | 割切 | 対象環境に USS が存在しない (SSOT ④) |

→ 7 機能のうち 4 機能採用。バイナリ・ボリューム・USS は技術制約で割切。

### D 群: ロック / Check-in/out (T12)

**対象機能:** Change Tracker (S15) のロック / Check-in/out 5 機能
**論点:** 同時編集事故の防止という運用要件を、z/OS 側強制力なしでどう満たすか

| # | 機能 | 判定 | 実装方法 / 割切理由 |
|---|---|---|---|
| D1 | CHECKOUT (チェックアウト) | 採用 | Excel ワークシートに擬似フラグ |
| D2 | CHECKIN (チェックイン) | 採用 | Excel ワークシートで戻し操作 |
| D3 | FORCED CHECKIN (強制リリース) | 採用 | Excel 管理者ボタン |
| D4 | RESERVED (予約状態) | 採用 | Excel ステータス列 |
| D5 | z/OS 側 ENQ ロック | 割切 | z/OS 内部のため z/OS 外から制御不能 |

→ 5 機能のうち 4 機能採用。z/OS 側 ENQ ロックは技術的に z/OS 外から制御不能のため割切。

### E 群: リカバリ (T13)

**対象機能:** Change Tracker (S15) のリカバリ 5 機能
**論点:** 過去版への戻しという運用要件を、書き戻しできない制約下でどう満たすか

| # | 機能 | 判定 | 実装方法 / 割切理由 |
|---|---|---|---|
| E1 | MEMBERRECOVER (メンバ復元) | 採用 | Excel から TXT 出力 → 人手で 3270 貼付 |
| E2 | ARCHIVERECOVER (アーカイブ復元) | 採用 | 別ブックから TXT 出力 → 人手貼付 |
| E3 | PARTIAL RECOVER (部分復元) | 採用 | Excel で部分行抽出 → 貼付 |

## §D. 技術実現性

### D-1. z/OS 出力ユーティリティ (T26)

本ツールが運用資材を取得するための z/OS 側出力ユーティリティを特定。

| IEHLIST | DFSMSdfp の標準ユーティリティ | PDS のディレクトリ情報（メンバ名・サイズ等）を出力 | メンバ一覧取得の主用途 | DFSMSdfp Utilities (S19) で PDS ディレクトリ出力を確認 |

### D-2. z/OS → Excel への取り込み経路 (T27)

PCOMM の IND$FILE による Shift-JIS テキスト転送。

| 手順 | 内容 | 依存 |
|---|---|---|
| 1. JCL 投入 | z/OS 上で IEBPTPCH/IEHLIST を実行し、シーケンシャルファイルに出力 | z/OS 標準機能 |
| 2. PCOMM 接続 | PC 側で PCOMM を起動、3270 セッションを z/OS にログオン | PCOMM Quick Beginnings (S20) |
| 3. IND$FILE 受信 | PCOMM の Receive File 機能で z/OS シーケンシャルを Shift-JIS 変換しつつ PC へ転送 | PCOMM 標準機能 (CONVERT=YES オプション) |
| 4. VBA 読込 | Excel/VBA で受信ファイルを Open For Input → Line Input で行単位読込 | VBA 公知機能 |
| 5. 履歴シートに反映 | VBA で履歴シートに行追加 → 前回版とテキスト比較 | 本ツール独自実装 |

→ PCOMM の IND$FILE で Shift-JIS テキスト転送、VBA で読み込み可能と判断。根拠強度=中（PCOMM 公式機能 (S20) と VBA テキスト処理の組合せ。実装段階での試作検証が必要）。性能 (T28: 数千メンバ規模での所要時間) は試作で実測する。

## §H. 反論への先回り

想定される反論 4 件と回答。

??? question "反論 1: Change Tracker のような公式ツールがあるなら、本ツールを作る必要はないのでは?"
    Change Tracker は priced feature + STC 常駐 + RACF プロファイル設計を要し、想定組織（利用者単独〜5 人未満）では現実的に導入できない。本ツールは Change Tracker の代替ではなく、Change Tracker を導入できない現場のための別製品。

??? question "反論 2: Git 等の既存 SCM を運用資材にも適用すれば良いのでは?"
    アプリ部門の SCM を運用資材まで拡張する案 (T8) は別検討事項。本資料は運用資材領域限定の検討であり、SCM 全面導入は本資料の射程外。SCM 拡張案は別途比較検討する余地がある。

??? question "反論 3: z/OS 側に何か置けば、もっと機能を実現できるのでは?"
    本ツールは SSOT ① で「z/OS 側に常駐プログラム・データセット・追加ライセンスを置かない」を前提としている。これは Change Tracker を導入できない現場の制約（z/OS 設置物への運用工数・ライセンス費用を許容できない）を反映した設計判断であり、前提を緩めることは別検討資料の対象。

??? question "反論 4: PCOMM 経由の取得は時間がかかるのでは?"
    T28（大規模ファイル性能）で試作実測の課題として明示済み。数千メンバ・数 MB 超では分割取得で対応する設計とする。標準的な運用資材（数十〜数百メンバ規模）では PCOMM 標準速度で実用範囲と試算。
